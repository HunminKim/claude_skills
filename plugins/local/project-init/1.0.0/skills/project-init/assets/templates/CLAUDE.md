## 개발 원칙 요약

### 1. 계획 우선
- 코드 작성 전에 반드시 `docs/development_plan.md` 작성
- 여러 파일 수정 시: 계획 → 사용자 승인 → 순차 실행
- 작업 전 요구사항/제약조건/입출력을 5~10줄로 요약

### 2. 소단위 작업
- 작업은 반드시 독립적으로 테스트 가능한 소단위로 분리
- 각 소단위 완료 시: 구현 → **@verifier 에이전트에게 검증 위임** → 결과 기록 → `docs/checklist.md` 업데이트
- 구현한 에이전트가 직접 검증하지 않는다. 반드시 verifier 서브에이전트를 통해 독립 검증한다
- 모든 작업 완료 후 사용자 최종 확인(완료 사인) 획득

### 3. 코드 규칙
- 문서/주석: 한국어 작성 (라이브러리명·API명은 영어 유지)
- 함수: 단일 책임, 50줄 이내, 명확한 네이밍 (축약어 금지)
- 함수 호출 깊이: 기본 1~2단계 (최대 3단계, 사유 명시)
- Early Return 사용, 중첩 반복문 최대 2단계
- 매직넘버 금지 → 상수화, 인덱스 매핑은 Enum+Dict 사용
- 정의부(상수/enum/schema) 확인 없이 코드 수정 금지

### 4. 에러 처리
- 빈 except 금지, 에러 메시지에 원인+상황 포함
- 로그 레벨: DEBUG(내부흐름) / INFO(주요이벤트) / ERROR(실패)
- 외부 입력 반드시 검증, 민감 정보 출력 금지

### 5. 보안 및 의존성
- 하드코딩 금지 → config/env 사용
- 파일 경로는 pathlib 사용
- 새 라이브러리 추가 시: 도입 이유·대안·선택 근거 명시

### 6. 디버깅
- 디버깅 발생 시 `docs/debug/YYYYMMDD_이슈명.md` 작성
- 패치 노트 포함: 문제 원인 / 수정 내용 / 영향 범위 / 재발 방지

### 7. 완료 후 절차
- 사용자 완료 사인 이후 전체 프로세스 상세 문서 작성

### 8. 기타
- 모든 시간 표기: KST 기준
- 최종 배포: Docker 기반, 환경 세팅 방법 별도 기록
